## 1. 作りたいもの（一行で）
**歩兵・装甲戦闘車両・ロボット・航空機・艦船** を運営する司令官になり、各兵科のユニットをアセンブル（HoI4 のデザイナー系参考）して出撃させる、**多兵科ミリタリー運営シム × メカアクション** のハイブリッドゲーム。

## 2. なぜ作るのか（モチベーション）
- 自分（ネットワークエンジニア／コードは書かない）が、Claude を使ってオリジナルゲームを完成させたい
- 既存のミリタリー／メカゲームで好きな要素を組み合わせた、自分が遊びたいゲームが市場にない
- AI 単独でどこまで作れるかの実験も兼ねる
- **最終的には他人に遊んでもらえる製品版クオリティを目指す**（個人用プロトタイプで終わらせない）

## 3. 開発体制
- **仕様策定・方向性決定**：claude.ai（このチャット）
  - このチャットでは**実装・コーディング・設計構築は行わない**
  - 仕様書（PLAN/SPEC/TODO/KNOWLEDGE）の作成と意思決定のみ
- **実装・設計構築**：**Claude Code と Codex の2つのみ**を使用する
  - Claude Code：ターミナルファースト、大きめのコンテキスト（約1Mトークン）、GDScript の構文ドリフトが少ない
  - Codex：複数サーフェス（CLI/IDE/アプリ/クラウド）、PRレビュー等に強い。コンテキストは約272Kトークン
  - **相互チェック運用**：一方が書いたコードを他方にレビュー・セカンドオピニオンさせ、コードを読めない弱点を補強する
- **自分の役割**：仕様確認、動作検証、方向性の判断（コードは読まない／書かない）
- **ドキュメント運用**：PLAN.md / SPEC.md / TODO.md / KNOWLEDGE.md の 4 点セット

## 4. 好きなゲームと取り入れたい要素

**世界観：現代軍事 ＋ SF メカ混在（MGS／ピースウォーカー型）** で確定。現代の歩兵・戦車・艦船・航空機が活躍する世界に、メタルギア級の巨大兵器や AC ／ガンダム的なメカが混在する。**時代設定は現代〜近未来兵器のみ**（WW2 以前の旧式兵器は登場しない）。トーンは「実在しそうな架空の現代軍事」（MGS／エースコンバットのストレンジリアル系）。

| 元ネタ | 取り入れる要素 |
|---|---|
| アーマード・コア | パーツ組み換えによる機体アセンブル |
| ピースウォーカー / TPP | 基地運営、兵士スカウト、研究開発、派遣任務、世界観のベース |
| ガンダム | 機体 × パイロットの組み合わせで性能変動、機体ごとの個性 |
| メタルギア（兵器） | ボス級巨大兵器との特殊戦 |
| 地球防衛軍 | 大量の敵を薙ぎ払う爽快感、武器コレクション |
| エースコンバット | 空戦任務、レーダー運用 |
| ハーツオブアイアン | 兵科デザイナーの構造、戦略性 |

## 5. ゲームの全体像

**2 つのモードを行き来する構造**：

```
[司令官モード]                [出撃モード]
基地運営・編成・研究      ⇄    機体を動かして実戦
（戦略性・じっくり）           （爽快感・テンポ良く）
```

### 5.1 司令官モード（戦略レイヤー）
- 全兵科のユニットアセンブル（5 兵科それぞれのスロット構造でパーツ組み換え）
- 人員管理（スカウト・配属。人員は単純な数として扱い、育成・成長要素は持たない）
- 研究開発（新パーツの解放）
- 資源管理（資金、素材、人員）
- 任務選択（次にどの出撃ミッションへ行くか）

### 5.2 出撃モード（戦術レイヤー）
- 組んだユニットをプレイヤーが**手動操作**して戦闘（兵科ごとに操作系は異なる）
- 通常任務：雑魚を多数撃破するミッション（EDF 的）
- 特殊任務：ボス級巨大兵器との戦闘（メタルギア的）
- 戦果に応じてパーツ・素材・搭乗員経験値を持ち帰る

### 5.3 ループ
司令官モードでユニットを組む → 出撃して戦う → 戦利品を持ち帰る → より強いユニットを組む → さらに難しい任務へ。

## 6. 何を「面白さの核」にするか
- **アセンブルの沼**：パーツ組み合わせでユニット性能が大きく変わる（全 5 兵科）
- **編成の戦略性**：諸兵科連合での運用、部隊単位の組み合わせ
- **任務の達成感**：自分で組んだユニットで強敵を撃破する
- **収集と拡張**：パーツ・搭乗員・ユニットデータを集めていく楽しさ

## 7. やらないこと（スコープ外）
- スマホ移植（PC 向け前提）
- PvP マルチプレイ（PvE のみ。詳細は §8.4 参照）

## 8. 品質目標（3 段階のクオリティ階段）

優先順位は「**自分が楽しい**」が最優先。それが満たされた上で順に積み上げる。

### 8.1 第1段階：自分が遊んで楽しい（最優先）
- ゲームの**コア要素そのもの**のクオリティを上げる
  - アセンブルが楽しい（パーツ組み合わせの奥行き、悩ましさ）
  - 戦闘が爽快／戦略的に深い
  - ループが回る（出撃 → 戦利品 → 強化 → さらに難しい任務）
  - バランスが破綻していない
- グラフィック・サウンド・UI は仮素材でも OK
- チュートリアルなし、自分が分かっていればいい
- **完了の判断は自分が「もうこのまま遊び続けられる」と感じた時**

### 8.2 第2段階：他人にも遊んでもらえる
第1段階が「いい感じ」になってから着手：
- チュートリアル／オンボーディング
- UI/UX 仕上げ（初見プレイヤーが迷わない）
- ヘルプ・ツールチップ・操作説明
- アセット差し替え（仮素材から本番素材へ）
- バグ修正の徹底
- キーコンフィグ等のカスタマイズ性

### 8.3 第3段階：商用配信（最終ゴール）
- 配布プラットフォーム対応（itch.io / Steam 等）
- ライセンス・著作権整理（特にアセット）
- 特商法・利用規約等の法的整備
- ローカライズ（必要なら英語対応）
- サポート窓口・バグ報告体制
- セーブデータ互換性ポリシー
- 体験版の有無

### 8.4 マルチプレイ方針（将来実装）
- **PvE 協力プレイ**（PvP はやらない）
- **ホスト間通信（P2P）方式**：専用サーバを立てず、誰か一人がホストになる
- **実装フェーズ**：Phase 5 以降の機能拡張として位置付ける
- **設計上の制約（重要）**：シングルプレイ実装の時点から「ネットワーク同期可能なデータ構造／状態管理」で組む。後付けは事実上の作り直しになるため
- ネットワーク設計はあなた（ネットワークエンジニア）の本職領域なので強み

### 8.5 3D 化方針 ★決定
- **採用方針：2D 固定（3D は次回作として割り切る）**
- 判断理由：
  - 製品版クオリティでの完成が最優先
  - 5 兵科 × 区分 × アセンブル × マルチプレイの現スコープだけで既に大規模
  - 3D 採用時はモデル・テクスチャ・アニメ・物理・カメラの全てが必要
  - Claude 単独・非プログラマ体制では 2D でも十分挑戦的
  - 2D で完成 → 続編で 3D 化のルートは残る（逆順は不可能）
- カメラアングルはトップダウン / アイソメトリック等から後で選定（技術スタック決定後）
- 3D 化を再検討する場合は PLAN/SPEC を改訂して明示する

## 9. 現時点の不安・懸念
- 出撃モード（リアルタイム手動操作）が技術的に一番難しい領域
- スコープが大きいため、最後まで完成させられるかが最大のリスク
- 自分でコードが読めないため、Claude の出力が正しいか検証する手段を確立する必要がある
- 製品版クオリティ要求がスコープを大幅に押し上げる
- アセット（絵・音）の調達方法が未決定（フリー素材 / AI 生成 / 自作 / 外注）
- **マルチプレイ後付けの罠**：シングル前提で組むと後から同期対応は事実上の作り直し

## 10. 段階リリース方針

3 段階のクオリティ階段に沿って Phase を組む。

**【第1段階：自分が遊んで楽しい】**
1. **Phase 1**：1 兵科のみで最小構成（兵科は SPEC.md で検討中）
2. **Phase 2**：出撃モード（手動操作）を追加 — 自分が触りたい中核要素
3. **Phase 3**：他兵科追加、研究開発の拡張
4. **Phase 4**：コアゲームのクオリティアップ — バランス調整、コンテンツ追加、面白さの深掘り
   - **Phase 4 完了の判断は自分が「このまま遊び続けたい」と思えた時**

**【第2段階：他人にも遊んでもらえる】**
5. **Phase 5**：他人向け機能 — チュートリアル、UI/UX 仕上げ、アセット差し替え、ヘルプ整備、バグ修正の徹底

**【第3段階：商用配信（最終ゴール）】**
6. **Phase 6**：配信準備 — ライセンス整理、配布プラットフォーム対応、QA、ローカライズ、法的整備

詳細は `SPEC.md` で詰める。
