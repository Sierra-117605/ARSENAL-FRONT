# ARSENAL FRONT — SPEC.md（全兵科共通仕様）

> **プロジェクト名：ARSENAL FRONT**
> リポジトリ：https://github.com/Sierra-117605/ARSENAL-FRONT

## このドキュメントの位置づけ
- 兵科（歩兵・車両・ロボット・航空機・艦船）を問わず共通で使う仕様
- Phase 1 でどの兵科を選んでも、ここの仕様は使い回せる
- **決定事項** と **[TBD] 未決定事項** を明確に分けて記載
- 未決定事項は選択肢だけ並べ、勝手に確定させない
- **このドキュメントは仕様策定のみ。実装は Claude Code と Codex の2ツールで行う**
- **品質目標は 3 段階で積み上げる**：
  1. 自分が遊んで楽しい（最優先）
  2. 他人にも遊んでもらえる
  3. 商用配信（最終ゴール）

## 1. 決定事項（議論済み）

### 1.1 ゲームの全体構造
- **司令官モード（戦略レイヤー）**：基地運営・編成・研究・任務選択
- **出撃モード（戦術レイヤー）**：プレイヤーが機体を**手動操作**して戦闘
- 両モードを往復するループ

### 1.2 兵科と世界観
- **世界観：現代軍事 ＋ SF メカ混在（MGS／ピースウォーカー型）**
- **時代設定：現代〜近未来兵器のみ**。WW2 以前の旧式兵器（駆逐戦車・レシプロ機・戦艦主砲主体の艦隊等）は登場しない。実在の現代兵器をベースに、レールガン・レーザー・SF メカ等の近未来要素を上乗せする
- **質感・トーン：「実在しそうな架空の現代軍事」**。MGS シリーズ、エースコンバットのストレンジリアル系の雰囲気。冷戦的な緊張感、無機質だが重厚なミリタリー美学。過度にダークでも無味乾燥でもない中間
- 歩兵 / 装甲戦闘車両 / ロボット / 航空機 / 艦船 の **5 兵科**
- 全兵科に **アセンブル（デザイナー）機能** をつける（HoI4 のデザイナー系を参考）
- **各兵科の中に「区分」を設ける**（次節 1.3 参照）

### 1.3 兵科 × 区分体系
ひとくくりの兵科でもユニット種類によりスロット構造が違う（例：戦闘機と爆撃機）ため、**兵科の中に「区分」を設ける**。

```
ユニット = 兵科 + 区分 + スロット構造（区分ごと） + 装着パーツ + ステータス計算
```

- 区分が違えば**選べるスロットの種類・数が違う**
- 区分自体は固定（プレイヤーが「これは戦闘機」と決めて設計を始める）
- 同じパーツでも区分ごとに使える／使えないが分かれる場合あり

具体的な区分候補は §1.4 に列挙（詳細仕様は [TBD]）。

### 1.4 ユニットの統一構造
すべての兵科は同じ枠組みで扱う：

```
ユニット = 兵科 + 区分 + スロット定義 + 装着パーツ + ステータス計算式
```

- スロットの種類・数は **兵科 × 区分** ごとに違う
- パーツのカタログも兵科 × 区分に応じて選択肢が変わる
- **計算ロジックは共通基盤として 1 つだけ実装する**

**区分の候補一覧（MGS 型世界観想定、詳細は [TBD]）：**

| 兵科 | 区分例 | スロット例 |
|---|---|---|
| 歩兵 | 一般歩兵 / 機甲歩兵 / 空挺 / 特殊部隊 / 対戦車 / 工兵 / 衛生兵 / 偵察 | 主武器・副武器・装備・分隊編成 |
| 車両 | 主力戦車(MBT) / 偵察戦闘車 / 対戦車ミサイル車両 / IFV / APC / 自走砲 / 自走対空 / 装輪装甲車 | 車体・砲塔・装甲・エンジン・特殊モジュール |
| ロボット | 軽量機 / 中量機 / 重量機 / 二足 / 四足 / ホバー / 飛行型 / メタルギア級巨大兵器 | 頭・胴・腕・脚（または推進系）・武器・ジェネレータ |
| 航空機 | 戦闘機 / 攻撃機 / 爆撃機 / 偵察機 / 輸送機 / 攻撃ヘリ / 輸送ヘリ / AWACS / UAV | 機体・エンジン・武装・アビオニクス |
| 艦船 | 駆逐艦 / 巡洋艦 / 戦艦 / 空母 / 強襲揚陸艦 / 潜水艦 / 補給艦 | 船体・主砲・副砲・対空・装甲・機関・艦載機格納 |

→ 各区分のスロット定義詳細は [TBD]（§2.16 参照）

### 1.5 段階リリース方針
1. Phase 1：**装甲戦闘車両**のみで最小構成を完成（§1.6 参照）
2. Phase 2：出撃モード（手動操作）追加 or 兵科追加（ロボット有力）
3. Phase 3 以降：拡張

### 1.6 Phase 1 詳細 ★決定
**Phase 1 で扱う兵科：装甲戦闘車両**

**採用理由：**
- ステータス体系（§2.1 の 8 項目）は元々戦車表現のための HoI4 由来。決めた体系の検証に最適
- HoI4 タンクデザイナーという明確な参考実装があり仕様がブレない
- 区分ごとの役割が現実準拠で分かりやすく、開発者が検証しやすい
- ここで作る共通基盤（ユニット＝区分＋スロット＋パーツ＋8ステータス計算）を他兵科へ流用できる
- 一番好きなロボットは基盤が固まった後の Phase 2〜3 に「ご褒美」として温存（最初の兵科は試行錯誤で作り直しが多いため）

**Phase 1 のゴール像：**
- 車体・砲塔・装甲・エンジン・モジュールを選んで戦車を組む（HoI4 タンクデザイナー的）
- 組んだ戦車のステータスが 8 項目（初期は 5 項目）で算出される
- 簡単な自動戦闘で性能差が確認できる
- ※出撃モード（手動操作）は Phase 2。Phase 1 は司令官モード＋自動戦闘まで

**Phase 1 で採用する車両区分（最小セット）：**
- まず **主力戦車(MBT) / 偵察戦闘車 / 対戦車ミサイル車両** の 3 区分程度から始める（[TBD]：この 3 つで良いか、増減するか）
- IFV / APC / 自走砲 / 自走対空 / 偵察車両 は Phase 1 完成後に追加
- 各区分のスロット構造詳細は §2.16 で確定する

**モチベ管理メモ：**
- 情熱がロボットにあるため「早くロボを作りたい」問題が出る可能性
- その場合 Phase 2 をロボットに前倒しする選択肢を残す

## 2. [TBD] 未決定事項

### 2.1 ステータス体系 ★決定
**候補 B（HoI4 風分離型）の簡略版**を採用。全 8 項目を全兵科共通の土台とする。

**攻撃系**
- **対人攻撃（Soft Attack）**：装甲の薄い目標（歩兵・非装甲車両）への火力
- **対装甲攻撃（Hard Attack）**：装甲の厚い目標（戦車・艦船）への火力
- **貫通**：相手の装甲を抜けられるか

**防御系**
- **耐久（HP）**：撃破までの耐久値
- **装甲**：貫通されなければダメージを激減させる

**運用系**
- **機動**：回避・移動・行動順に影響
- **索敵**：相手を発見する／されにくくする範囲
- **射程**：どの距離から攻撃できるか

**採用理由：**
- 兵科ごとの三すくみ（対人特化 ⇔ 装甲特化 ⇔ 貫通特化）が数値で自然に成立し、諸兵科連合が意味を持つ
- 全 5 兵科に無理なく適用できる（AC 風の EN 概念は入れない）
- アセンブルにトレードオフが生まれる（対装甲を上げると対人が下がる等）
- 後から「対空」「制空」「ステルス」等を拡張可能

**段階導入メモ：**
- 実装初期（Phase 1 プロトタイプ）は **対人 / 対装甲 / 貫通 / 耐久 / 装甲 の 5 項目**で動かしてよい
- 機動 / 索敵 / 射程 は出撃モード実装時に追加する（[TBD]：このタイミングで良いか）

### 2.2 ダメージ計算式 ★方向決定（詳細係数は [TBD]）
ステータス体系（2.1）に合わせ、**HoI4 風の貫通式**を採用する。

**基本ロジック：**
1. 攻撃側は目標の装甲量に応じて **対人攻撃** か **対装甲攻撃** のどちらが効くかが変わる
   - 目標が「非装甲」寄り → 対人攻撃が主に通る
   - 目標が「装甲」寄り → 対装甲攻撃が主に通る
2. **貫通 ≧ 目標の装甲** ならフルダメージ、**貫通 < 装甲** ならダメージ大幅減衰
3. 上記からダメージを算出し、耐久（HP）を削る

**[TBD] 詳細：**
- 減衰率の具体的な数値（貫通不足時に何 % になるか）
- 対人/対装甲の切り替えを「閾値で二択」にするか「装甲量に応じた連続変化」にするか
- クリティカル・弱点判定の有無
- 部位ダメージの有無（巨大兵器のボス戦では部位破壊があると面白い → [TBD]）

→ 具体的な数値バランスは実装しながら調整（KNOWLEDGE.md に知見を蓄積）

### 2.3 戦闘ルールの大枠 ★決定（詳細数値は [TBD]）

戦闘ルールは 2 層に分かれる。Phase 1 で実装するのは司令官モード側＋自動戦闘。

#### A. 司令官モード（戦略レイヤー）の時間進行 ★決定
- **ターン単位：日単位**
- 1 日（1 ターン）でできること：車両の設計・製造、研究の割り当て、任務への部隊派遣、資源のやりくり
- 「日を進める」と時間が経過し、派遣中の任務が解決され、研究・製造が進む
- **作業感を防ぐ仕組み（必須）**：何もない日を手動で連打させない。次にイベント（任務完了・研究完了・製造完了等）が起きる日まで自動で時間を進める「スキップ進行」を用意する（HoI4 方式）
- [TBD]：1 日あたりの研究・製造の進み速度、任務の所要日数レンジ

#### B. 出撃（戦術レイヤー）の解決方法 ★決定（Phase 1）
- **Phase 1 は自動戦闘（候補 2：ターン制シミュレーション型）を採用**
- ユニットが 1 体ずつ行動し、命中・ダメージを逐次処理してログを出す（例：「主力戦車 A が敵 IFV を撃破」）
- 採用理由：自分で組んだ車両が戦闘ログ上で活躍するのが見え、アセンブルの良し悪しが直感的に分かる。Phase 2 の手動出撃モードへの布石にもなる
- 手動操作は Phase 2 で追加

#### C. 戦闘で参照するルール（候補 2 前提）
- ユニットは 8 ステータス（§2.1）を持ち寄って戦う
- 行動順は **機動** が高い順
- 攻撃時、目標の装甲に応じて対人攻撃／対装甲攻撃が選択され、**貫通 vs 装甲**で命中後ダメージが決まる（§2.2）
- **射程** が長いユニットは先制攻撃できる
- **索敵** で先に発見した側が有利を取る
- 耐久が 0 になったユニットは撃破
- 片軍全滅 or 任務目標達成で決着
- [TBD]：命中率の計算式、先制攻撃の具体ルール、複数ユニット同士の targeting ロジック

#### D. Phase 1 でやらないこと（スコープ確定）
- 手動操作（Phase 2）
- 地形効果（後で追加）
- 諸兵科連合の連携ボーナス（他兵科が増えてから）
- ボス戦（Phase 3）

#### E. 出撃モード（手動操作）詳細 [TBD]（Phase 2 で確定）
- 視点（トップダウン / アイソメトリック / サイドビュー）
- 操作系（キーボード / ゲームパッド対応）
- ステージ構造（オープン / ミッション固定マップ）

### 2.4 資源体系 ★Phase 1 暫定確定（拡張余地あり）

**Phase 1 暫定決定**：**資金 (funds) と 素材 (materials) の 2 種類で開始**する。

**採用理由：**
- 「製造 → 出撃 → 報酬」の最小ループはこの 2 種類で成立する
- 研究ポイントと人員を最初から入れると UI が膨らみ、Phase 1 ゴールから外れる

**初期値（仮、調整余地あり）：**
- 資金（funds）：1000
- 素材（materials）：500

**製造コスト計算式（仮）：**
- 資金コスト = ユニット重量 × 10
- 素材コスト = ユニット重量 × 5

### 2.8 技術スタック ★決定
- **Godot 4（2D モード）+ GDScript** を採用

**判断理由：**
1. 製品版クオリティの UI が標準ツール（Control ノード）で構築可能
2. P2P マルチプレイ API が標準装備（PvE × P2P の将来要件と整合）
3. 開発者自身がコードを読み書きしないため、Python 縛りの実利が薄い
4. シーン管理により 5 兵科 × 区分の規模に耐える構造
5. 続編で 3D 化する際にエンジン乗り換え不要

**留意点：**
- 自分（開発者）が Godot エディタを少しだけ触る場面はある（プロジェクト起動・書き出し等）
- Claude/Claude Code の GDScript 出力品質は Pygame よりトレーニング情報量が少ない可能性。実装初期に検証が必要。Claude Code と Codex の相互チェックで補強する

### 2.9 [TBD] アセット調達方針
3 段階の品質階段ごとに方針が変わる：

- **第 1 段階（自分が遊ぶ）**：仮素材で OK（フリー素材、図形、テキストのみでも可）
- **第 2 段階（他人にも遊ばせる）**：本番素材に差し替え
- **第 3 段階（商用配信）**：ライセンスを完全に整理した素材のみ

**本番素材の候補：**
- **A: フリー素材中心**（OpenGameArt、itch.io 等から CC0/CC-BY を集める）
  - 安価、即着手可、世界観が散らかりやすい
- **B: AI 生成中心**（画像生成 AI で量産）
  - スタイル統一しやすい、商用ライセンス・著作権リスクの整理が必要
- **C: 自作**
  - クオリティ次第、時間がかかる、絵心の問題
- **D: 外注**（クラウドソーシング等）
  - 品質確保しやすい、コストが発生
- **E: A〜D のハイブリッド**

→ 配布方針（2.10）と連動して決定。第 1 段階では深く考えなくて良い

### 2.16 [TBD] 兵科 × 区分の詳細スロット定義
§1.4 の表は候補。実装にあたっては各区分ごとに以下を確定する必要がある：

- **採用する区分**（候補から削るもの／追加するものはあるか）
- **区分ごとのスロット構造**（スロット名・数・必須/任意・容量上限）
- **区分間で共有するパーツ**と**区分専用パーツ**の切り分け
- **区分間の役割分担**（例：偵察戦闘車は偵察重視、主力戦車は装甲重視）
- **メタルギア級巨大兵器の扱い**（プレイヤーが設計可能？ボス専用？両方？）
- **空母の艦載機格納と航空機の連動**（艦船で出撃 → 艦載機を発艦 → 航空機モードへ、のような動線）

→ **Phase 1 の装甲戦闘車両については §2.18 で確定済み**。他兵科は各 Phase 着手時に確定する

### 2.18 Phase 1 装甲戦闘車両 スロット定義 ★決定
HoI4 タンクデザイナー並みの粒度。「固定 6 スロット ＋ 可変モジュールスロット」構成。設計方針はリアルとゲーム性の両立（スロット名・パーツ名は実在準拠、数値はゲーム用に調整）。

**固定スロット（各 1 つ必須選択）：**

| スロット | 選択肢の例 | 主な影響ステータス |
|---|---|---|
| 車体（Hull） | 軽量 / 中量 / 重量 | 搭載容量・基礎耐久・基礎装甲・モジュール枠数・コストの土台 |
| 主砲（Main Armament） | 滑腔砲（中口径/大口径）/ 機関砲 / 対戦車ミサイルランチャー / 榴弾砲 / ［近未来］レールガン / ［近未来］レーザー砲 | 対人攻撃・対装甲攻撃・貫通・射程 |
| 砲塔（Turret） | 旋回砲塔 / ミサイルランチャー架 / 無人砲塔（RWS）/ 無砲塔固定 | 旋回性能・命中・搭載可能な主武装サイズ上限 |
| エンジン（Engine） | 低出力 / 標準 / 高出力 | 機動（重量比で決定） |
| 装甲（Armor） | 軽装甲 / 中装甲 / 重装甲 / 傾斜装甲 | 装甲・耐久（重量増 → 機動低下） |
| サスペンション（Suspension） | 標準 / 高速型 / 重荷重型 | 機動・搭載重量上限 |

**可変モジュールスロット（車体サイズに応じて装着可能数が変動）：**
- 副武装（機銃 / 対空機銃）
- 追加装甲
- 無線機（索敵・連携）
- 暗視装置（索敵）
- 自動装填装置（連射）
- 増加燃料
- 煙幕発射機
- 地雷除去ローラー
- ※車体が大きいほどモジュール枠が増える＝重く高価になるトレードオフ

**3 区分のスロット傾向（同じ枠・中身が変わる）：**
- **主力戦車（MBT）**：旋回砲塔＋中〜大口径砲＋中〜重装甲＋標準サス。全スロット高水準の優等生。耐久◎装甲◎対装甲◎ / 機動△
- **偵察戦闘車**：軽量車体（装輪／装軌は選択可）＋小口径砲または機関砲＋軽装甲＋高速サス＋高出力エンジン。モジュールに索敵装備。機動◎索敵◎ / 装甲△対装甲△
- **対戦車ミサイル車両**：対戦車ミサイルランチャー搭載＋軽〜中装甲。遠距離から高貫通の一撃で戦車を仕留める待ち伏せ型。対装甲◎◎貫通◎◎射程◎ / 装甲△近接戦×。WW2 駆逐戦車の現代版に相当

- 各パーツの具体的な数値
- 重量とモジュール枠数の上限ルール
- パーツのアンロック条件（研究との連動）
- 区分専用パーツと共通パーツの線引き
