# Claude Code ⇄ Codex 相互レビュー運用

> このドキュメントは TODO T1-3 の運用ルールです。
> 根拠：PLAN §3（実装体制）、SPEC §2.8（GDScript 出力品質の留意点）、KNOWLEDGE §2-6（コードを読めない弱点を補強）。

## 1. なぜ相互レビューが必要か

- 開発者本人はコードを読めない／書けない → 実装の正しさを人間が直接確認できない
- 「動かして結果を見る」だけでは、隠れたバグやデータ構造の罠は見逃しやすい
- そこで **Claude Code と Codex の2ツール** で書く側と見る側を分け、お互いに指摘し合うことで品質を担保する

## 2. 基本フロー（1 タスクあたり）

```
1. Claude Code が実装（または Codex が実装）
   ↓
2. 反対側のツールにレビュー依頼（次節のテンプレ）
   ↓
3. 指摘事項を実装者が修正
   ↓
4. ユーザーが「動かして」結果確認
   ↓
5. 完了。学びは KNOWLEDGE.md へ
```

## 3. レビュー依頼の書き方（テンプレ）

レビューを頼むときは、相手にこの形で渡す：

```
■ レビュー依頼

【タスク】 TODO T?-?（例：T1-1 Godot プロジェクト作成）
【関連仕様】 SPEC §?.?
【実装したもの】 (変更したファイルのパス一覧)
【特に見てほしい点】
- (1) 仕様と矛盾していないか
- (2) シリアライズ可能データ構造の方針（docs/DATA_STRUCTURE.md）に沿っているか
- (3) Godot 4 / GDScript として正しい書き方か（古い構文・非推奨API混入なし）
- (4) Phase 1 のスコープ外の機能を勝手に作り込んでいないか
【検証方法】 (動かして何を確認したか／頼みたいか)
```

## 4. レビューする側のチェックリスト

レビュー側はこの観点で見る：

- [ ] **仕様準拠**：PLAN.md / SPEC.md / TODO.md と矛盾していないか
- [ ] **スコープ**：Phase 1 のゴール（TODO.md 冒頭）を超えていないか
- [ ] **データ構造**：`docs/DATA_STRUCTURE.md` の大原則を破っていないか
  - 状態に Node や Vector2 を直接保存していないか
  - 状態更新が apply(event) 経由になっているか
- [ ] **Godot 4 構文**：`@onready`, `func _ready() -> void:` など Godot 4 系の書き方か
- [ ] **ハードコード禁止**：パーツ数値などはコード直書きせず `data/*.json` から読んでいるか
- [ ] **ファイル配置**：`scenes/`, `scripts/`, `data/`, `saves/` の分離を守っているか
- [ ] **テスト可能性**：ステータス計算など、ユーザーが手計算で検証できる粒度か

## 5. 詰まったときの記録

- 詰まり：`TODO.md` の該当タスクを `[!]` にする
- 原因・解決：`KNOWLEDGE.md` の §3 に追記
- 仕様の不明点が見つかったら：勝手に決めず `SPEC.md` の [TBD] に選択肢として残してユーザーに相談

## 6. 最初の練習タスク（T1-3 本体）

T1-3 を「終わり」にする条件：

1. Claude Code が作った最小サンプル（`scenes/main.tscn` + `scripts/main.gd`）を Codex にレビュー依頼する
2. Codex の指摘を踏まえて、Godot 4 として問題ない最小構成にする
3. ユーザーが Godot エディタで開いて F5 で起動し、画面に "ARSENAL FRONT" のタイトル文字が出ることを確認
